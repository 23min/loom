# CLAUDE.md

This repository uses [aiwf](https://github.com/23min/aiwf) to track planning state.

## Quick reference

- `aiwf check` — validate the planning tree.
- `aiwf add <kind> --title "..."` — create an entity (epic, milestone, adr, gap, decision, contract).
- `aiwf promote <id> <status>` — advance status.
- `aiwf history <id>` — show what happened to an entity.

The pre-push hook runs `aiwf check` automatically; broken state cannot be pushed.

Skills under `.claude/skills/aiwf-*/` are gitignored and regenerated on `aiwf update`.

## Engineering principles (the code-quality bar)

All loom code is held to [`docs/engineering/principles.md`](docs/engineering/principles.md) — the properties of code a senior engineer can pick up in hours, not weeks. Use it as the **prompt list** while writing and the **review checklist** before committing. The principles are forces, not rules; when they conflict, judgment wins, and each has an "overdone" failure mode (see the anti-patterns table).

For loom specifically, these are **load-bearing** — treat them as defaults, not aspirations:

- **B2 schemas at boundaries** + **D2 equivalence tests at seams** — loom emits findings that aiwf (and other consumers) read across a process boundary. Declare that JSON shape once, validate it on read, and test the loom-writer ↔ consumer-reader pair against shared scenarios.
- **C3 atomic writes** + **C1 single source of truth** — verification results and any persisted state are written atomically (temp + rename), with one canonical owner per fact.
- **E3 audit trail** + **G3 observable** — every verification decision records what was checked, the inputs it saw, and why it went the way it did (aiwf already models this for planning; loom's runtime should match it).
- **G1 reproducible** — same inputs → same outputs; push time/randomness/env to the edges. The verifier's nondeterminism (Z3) is isolated and surfaced, never silently folded into a result.

Start from these when writing the loom-ultralight harness (M-0001) and beyond.
